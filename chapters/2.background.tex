%================ Section 2.1: FITT-VP ==================
\section{หลักการออกแบบโปรแกรมออกกำลังกาย (FITT-VP Principle)}
\label{sec:fitt-vp}

\noindent
หลักการ FITT-VP เป็นแนวทางมาตรฐานในการออกแบบโปรแกรมการออกกำลังกายให้เหมาะสมกับแต่ละบุคคล \cite{acsm2021} โดยประกอบด้วยองค์ประกอบสำคัญ 6 ประการ ได้แก่

\begin{itemize}
  \item \textbf{Frequency (ความถี่)}: จำนวนครั้งในการออกกำลังกายต่อสัปดาห์
  \item \textbf{Intensity (ความหนัก)}: ระดับความเข้มข้น เช่น เบา ปานกลาง หรือหนัก
  \item \textbf{Time (ระยะเวลา)}: ระยะเวลาในแต่ละครั้งที่ออกกำลังกาย
  \item \textbf{Type (ประเภท)}: รูปแบบของการออกกำลังกาย เช่น คาร์ดิโอ เวทเทรนนิ่ง
  \item \textbf{Volume (ปริมาณรวม)}: ปริมาณการออกกำลังกายรวม เช่น จำนวนเซตหรือเวลารวมต่อสัปดาห์
  \item \textbf{Progression (การเพิ่มระดับ)}: การปรับเพิ่มความยากหรือปริมาณอย่างเหมาะสม
\end{itemize}

ในการพัฒนา GymMate หลักการนี้ถูกนำมาใช้ในการแนะนำโปรแกรมการออกกำลังกายที่เหมาะสมกับผู้ใช้แต่ละคน เพื่อเพิ่มประสิทธิภาพและลดความเสี่ยงในการบาดเจ็บ

%================ Section 2.2: Self-Efficacy ==================
\section{ทฤษฎีแรงจูงใจและความต่อเนื่องในการออกกำลังกาย (Self-Efficacy \& Exercise Adherence)}
\label{sec:self-efficacy}

\noindent
ทฤษฎี Self-Efficacy ของ Bandura กล่าวถึงความเชื่อมั่นของบุคคลในความสามารถของตนเอง ซึ่งมีผลโดยตรงต่อพฤติกรรมและความพยายามในการทำกิจกรรมต่างๆ \cite{bandura1977} ในบริบทของการออกกำลังกาย

\begin{itemize}
  \item ผู้ที่มี self-efficacy สูง จะมีแนวโน้มออกกำลังกายสม่ำเสมอ
  \item ผู้ที่มี self-efficacy ต่ำ มักเลิกกลางคัน
\end{itemize}

\noindent
\textbf{Exercise Adherence} คือความสามารถในการรักษาพฤติกรรมการออกกำลังกายอย่างต่อเนื่องในระยะยาว \cite{hhs2018} ระบบ GymMate จึงออกแบบให้มีระบบติดตามความคืบหน้า การตั้งเป้าหมาย และ feedback แบบ real-time เพื่อเสริมสร้างแรงจูงใจ

%================ Section 2.3: Gamification ==================
\section{Gamification และการออกแบบระบบแรงจูงใจดิจิทัล}
\label{sec:gamification}

\noindent
Gamification คือการนำองค์ประกอบของเกม เช่น คะแนน (points), เหรียญรางวัล (badges) และอันดับ (leaderboard) มาใช้ในระบบที่ไม่ใช่เกม เพื่อเพิ่มแรงจูงใจของผู้ใช้ \cite{deterding2011}

ในระบบ GymMate มีการประยุกต์ใช้ Gamification ได้แก่

\begin{itemize}
  \item ระบบ \textbf{Leaderboard} จัดอันดับผู้ใช้งาน
  \item ระบบ \textbf{Achievement / Badge} มอบรางวัลจากการทำสำเร็จ
  \item ระบบ \textbf{Streak} วัดความต่อเนื่องของการออกกำลังกาย
  \item ระบบ \textbf{XP และ Level} สะสมประสบการณ์และพัฒนาระดับ
\end{itemize}

แนวคิดนี้ช่วยให้ผู้ใช้รู้สึกสนุก ท้าทาย และมีเป้าหมาย ส่งผลให้เกิดการใช้งานอย่างต่อเนื่อง

%================ Section 2.4: React Native ==================
\section{การพัฒนา Mobile Application (React Native \& Expo)}
\label{sec:react-native}

\noindent
React Native เป็น Framework สำหรับพัฒนา Mobile Application แบบ Cross-platform \cite{reactnative} ที่สามารถใช้ JavaScript/TypeScript ในการพัฒนาได้ทั้ง iOS และ Android

Expo \cite{expo} เป็นเครื่องมือที่ช่วยลดความซับซ้อนในการพัฒนา เช่น
\begin{itemize}
  \item ไม่ต้องตั้งค่า environment ที่ซับซ้อน
  \item สามารถทดสอบแอปผ่านมือถือได้ทันที
  \item รองรับ API ต่างๆ เช่น กล้อง เซนเซอร์
\end{itemize}

การใช้ React Native ร่วมกับ Expo ทำให้สามารถพัฒนาแอป GymMate ได้รวดเร็ว ลดเวลาในการพัฒนา และดูแลโค้ดได้ง่าย

%================ Section 2.5: Vite + React ==================
\section{การพัฒนา Web Application (Vite + React SPA)}
\label{sec:vite-react}

\noindent
Vite \cite{vite} เป็นเครื่องมือสำหรับพัฒนา Frontend ที่มีความเร็วสูง โดยใช้หลักการของ modern bundler

React \cite{react} เป็น JavaScript Library สำหรับสร้าง UI แบบ Component-based

SPA (Single Page Application) คือแอปที่โหลดหน้าเพียงครั้งเดียว และใช้ JavaScript ในการเปลี่ยนเนื้อหา ทำให้โหลดเร็ว UX ลื่นไหล และลดการ reload หน้า

GymMate Web ใช้ Vite + React SPA เพื่อให้ได้ประสบการณ์ผู้ใช้ที่รวดเร็ว และเหมาะกับระบบ dashboard เช่น Admin และ Trainer Portal

\section{การออกแบบ Backend ด้วย Clean Architecture และ REST API}
\label{sec:clean-arch}

\subsection{Clean Architecture}
\label{subsec:clean-arch}

Clean Architecture ถูกเสนอโดย Robert C. Martin \cite{martin2017} โดยเป็น software architecture pattern ที่มุ่งเน้นการแยก business logic ออกจากรายละเอียดเชิงเทคนิคภายนอก ทั้ง framework, database และ UI โดยมีหลักการสำคัญ 2 ประการ ได้แก่

\begin{enumerate}
  \item business rules สามารถทดสอบได้โดยไม่ต้องพึ่งพา UI, database หรือ web server ใดๆ
  \item ส่วนประกอบภายนอกสามารถเปลี่ยนแทนได้โดยไม่กระทบต่อ business logic กลาง
\end{enumerate}

\begin{figure}[h]
  \centering
  \includegraphics[width=.80\linewidth]{chapters/Images/Clean Architecture.png}
  \caption{แผนภาพ Clean Architecture ที่ใช้ในการออกแบบ Backend ของ GymMate}
  \label{fig:clean-arch}
\end{figure}

องค์ประกอบหลักของ Clean Architecture ที่นำมาใช้ใน GymMate แบ่งออกเป็น 4 layer ดังนี้

\begin{enumerate}
  \item \textbf{Models Layer} — ชั้นในสุด เก็บ core entity เช่น \texttt{User}, \texttt{Exercise}, \texttt{Program}, \texttt{WorkoutLog}, \texttt{GamificationProfile} ไม่มี dependency กับ layer ใดๆ
  \item \textbf{Repository Layer} — กำหนด interface สำหรับการเข้าถึงข้อมูล โดย Usecase Layer รู้จักเพียง interface ไม่รู้จัก implementation จริง (เช่น GORM สำหรับ PostgreSQL, MinIO สำหรับ object storage)
  \item \textbf{Usecase Layer} — ชั้น business logic หลัก ไม่มี dependency กับ Gin framework หรือ HTTP เก็บ logic เช่น การคำนวณ Flex Substitute ด้วย Cosine Similarity, กฎการให้ XP และ Badge
  \item \textbf{Delivery Layer} — ชั้นนอกสุด รับ HTTP request ผ่าน Gin framework แปลงเป็น input ส่งต่อให้ Usecase และแปลง output กลับเป็น JSON response
\end{enumerate}

กฎสำคัญของ Clean Architecture คือ dependency จะไหลเข้าด้านในเท่านั้น ทำให้ GymMate สามารถพัฒนาและทดสอบ business logic ของแต่ละฟีเจอร์ได้อิสระจากการเปลี่ยนแปลงของ infrastructure

\subsection{REST API}
\label{subsec:rest-api}

REST (Representational State Transfer) เป็น software architecture style สำหรับการออกแบบระบบที่สื่อสารผ่าน HTTP โดย GymMate ใช้ RESTful API เป็นช่องทางสื่อสารหลักระหว่าง Backend กับ Mobile App และ Web CMS Dashboard

Constraints หลักของ REST ที่นำมาใช้ใน GymMate ได้แก่

\begin{enumerate}
  \item \textbf{Client-Server} — แยก UI ออกจาก data storage อย่างชัดเจน ทำให้ Mobile App และ Web CMS พัฒนาได้อิสระจาก Backend
  \item \textbf{Stateless} — server ไม่เก็บ session state ของ client แต่ละ request ต้องบรรจุข้อมูลที่จำเป็นทั้งหมดในตัวเอง
  \item \textbf{Uniform Interface} — ใช้ HTTP Methods ตามความหมายมาตรฐาน และ versioning ที่ \texttt{/api/v1/}
  \item \textbf{Layered System} — client ไม่จำเป็นต้องรู้ว่าติดต่อกับ server จริงโดยตรงหรือผ่าน intermediary เช่น Cloudflare Tunnel
\end{enumerate}

\subsection{Stateless Application และ JWT Authentication}
\label{subsec:stateless}

Stateless หมายถึงการออกแบบให้ server ไม่เก็บ state ของ session ผู้ใช้ไว้ในหน่วยความจำของตัวเอง ข้อดีได้แก่

\begin{itemize}
  \item \textbf{Scalability} — สามารถเพิ่ม server instance ได้โดยไม่ต้องกังวลเรื่อง session synchronization
  \item \textbf{Reliability} — หาก server instance หนึ่งล้มเหลว request ถัดไปสามารถส่งไปยัง instance อื่นได้ทันที
\end{itemize}

GymMate implement Stateless ผ่านกลไก \textbf{JWT (JSON Web Token) HS256} โดยมีรายละเอียดดังนี้

\begin{itemize}
  \item เมื่อผู้ใช้ login สำเร็จ server จะออก JWT ที่ลงนามด้วย secret key และส่งกลับในรูป HTTP-only cookie
  \item ใน request ถัดๆ ไป client แนบ JWT มาด้วยทุกครั้ง server จะตรวจสอบลายเซ็นและดึงข้อมูล \texttt{user\_id} และ \texttt{role} ออกจาก token โดยตรง
  \item HTTP-only cookie ป้องกัน JavaScript ฝั่ง client อ่าน token ได้โดยตรง ลดความเสี่ยงจาก XSS attack
\end{itemize}

%================ Section 2.7: Cosine Similarity ==================
\section{Cosine Similarity}
\label{sec:cosine}

\noindent
Cosine Similarity เป็น metric ทางคณิตศาสตร์ที่ใช้วัดความคล้ายคลึงระหว่าง vector สองตัว โดยคำนวณจากมุมระหว่าง vector ทั้งสองในปริภูมิหลายมิติ (multi-dimensional space) นิยามดังสมการที่ \ref{eq:cosine}

\begin{equation}
  \text{cosine similarity}(\mathbf{A}, \mathbf{B}) = \frac{\mathbf{A} \cdot \mathbf{B}}{\|\mathbf{A}\| \, \|\mathbf{B}\|}
  \label{eq:cosine}
\end{equation}

โดยที่ $\mathbf{A} \cdot \mathbf{B}$ คือ dot product ระหว่าง vector และ $\|\mathbf{A}\|$, $\|\mathbf{B}\|$ คือ Euclidean norm ของ vector แต่ละตัว ผลลัพธ์มีค่าอยู่ในช่วง $[0, 1]$ สำหรับ vector ที่มีค่าไม่ติดลบ

\begin{table}[h]
\centering
\begin{tabular}{@{}ll@{}}
\toprule
\textbf{ค่า Cosine Similarity} & \textbf{ความหมาย} \\
\midrule
1.0                     & คล้ายกันสมบูรณ์ (ชี้ทิศทางเดียวกัน) \\
0.5 -- 1.0              & คล้ายกันในระดับสูง \\
0.0                     & ไม่มีความคล้ายคลึงกันเลย \\
ต่ำกว่า 0.0             & ชี้ทิศทางตรงกันข้าม \\
\bottomrule
\end{tabular}
\caption{การตีความค่า Cosine Similarity}
\label{tab:cosine}
\end{table}

ใน GymMate Cosine Similarity ถูกนำมาใช้ในระบบ Flex Substitute โดยแปลงข้อมูลสัดส่วนการใช้งานกล้ามเนื้อ (muscle activation percentage) ของแต่ละท่าออกกำลังกายเป็น vector แล้วคำนวณความคล้ายคลึงเพื่อแนะนำท่าทดแทนที่เน้นกล้ามเนื้อกลุ่มเดียวกันมากที่สุด

%================ Section 2.8: Related Work ==================
\section{งานวิจัยและระบบที่เกี่ยวข้อง}
\label{sec:related-work}

\noindent
มีงานวิจัยจำนวนมากที่ศึกษาการใช้เทคโนโลยีเพื่อส่งเสริมการออกกำลังกาย เช่น แอปติดตามสุขภาพ (Fitness Tracking Apps), ระบบแนะนำโปรแกรมออกกำลังกาย และระบบ Gamification เพื่อเพิ่ม motivation \cite{hhs2018}

ผลการศึกษาพบว่า การใช้ Gamification และระบบติดตามผลสามารถเพิ่มอัตราการออกกำลังกายอย่างต่อเนื่องได้ นอกจากนี้ยังมีแอปพลิเคชันที่คล้ายคลึง เช่น MyFitnessPal, Nike Training Club และ Strava ซึ่งเป็นแนวทางในการพัฒนา GymMate ให้มีฟีเจอร์ที่ตอบโจทย์ผู้ใช้มากขึ้น

%================ Section 2.9: Development Tools ==================
\section{เครื่องมือที่ใช้ในการพัฒนา}
\label{sec:dev-tools}

\noindent
ทีมพัฒนาได้เลือกใช้เครื่องมือที่ครอบคลุมทุกขั้นตอนของ software development lifecycle ตั้งแต่การวางแผน การออกแบบ การพัฒนา ไปจนถึงการ deploy และ monitoring โดยแบ่งตามหน้าที่ได้ดังนี้

\begin{itemize}
  \item \textbf{Project Management} — Jira สำหรับวางแผนและติดตามงานในรูปแบบ Agile โดยแบ่งงานออกเป็น Sprint จัดการ Backlog และกำหนด task ให้สมาชิก
  \item \textbf{UI/UX Design} — Figma สำหรับออกแบบ Wireframe, Prototype และ UI ของทั้ง Mobile App และ Web CMS Dashboard รองรับการทำงานร่วมกันแบบ real-time
  \item \textbf{Version Control} — Git และ GitHub โดยแบ่ง repository ออกเป็น 3 ส่วนตามชั้น architecture ได้แก่ BACKEND, FRONTEND และ MOBILE
  \item \textbf{CI/CD} — GitHub Actions สำหรับ pipeline อัตโนมัติ ได้แก่ CI pipeline (lint + unit test) และ CD pipeline (build Docker image → push → deploy อัตโนมัติ) พร้อม Discord Webhook แจ้งเตือนผลการ deploy
  \item \textbf{API Documentation} — Swagger UI สร้างจากไฟล์ \texttt{openapi.yaml} deploy ที่ port 8081 ใช้เป็นสื่อกลาง API specification ระหว่าง Backend กับทีม Frontend/Mobile
  \item \textbf{Infrastructure} — Docker + Docker Compose สำหรับ containerize application ทุกส่วน; Proxmox VE (Type 1 Hypervisor) สำหรับแบ่ง Virtual Machine; Tailscale (WireGuard-based VPN) สำหรับเชื่อมต่อ dev machine ทั้ง 3 เครื่องกับ server; Cloudflare Tunnel สำหรับ expose ระบบสู่ internet ที่ domain \texttt{gymmate.site} และ \texttt{api.gymmate.site}; Dozzle สำหรับ real-time container log viewer
\end{itemize}
